## Executive Summary

This audit examined twelve security-critical questions spanning key management,
fund flow integrity, access control, data privacy, database security, contract
upgradeability, concurrency, and network configuration. The platform's core
security design is sound — funds flow through an immutable on-chain escrow
contract, the deployer private key is never committed, and MCP tool authorization
correctly scopes mutations to the originating agent.

Three issues must be resolved before mainnet deployment:

1. **Supabase RLS policies leak PII** (notification channel addresses, waitlist
   emails) to anyone with the client-side anon key.
2. **No database-level immutability** on funded task records — a future code bug
   could silently modify the contractor wallet or payment amount.
3. **No transaction nonce management** — concurrent on-chain calls from the
   platform signer will fail under load.

None of these are actively exploitable on testnet today, but all three become
real risks at mainnet scale.

## Findings

### Critical / High

#### AUD-001: Notification channels and waitlist tables leak PII via anon key

**Severity:** High
**Category:** Data Privacy / RLS Misconfiguration
**Location:** `supabase/migrations/002_notification_channels.sql`, `supabase/migrations/004_waitlist.sql`

**Description:**
The `notification_channels` table has a `FOR SELECT TO anon USING (true)` policy
that was never revoked by migration 003 (which only dropped write policies). The
`waitlist` table was created in migration 004 with the same permissive read
policy.

Anyone who extracts the `SUPABASE_ANON_KEY` from client-side JavaScript can
query:
- `notification_channels.address` — email addresses, Telegram chat IDs, Discord
  user IDs, webhook URLs
- `waitlist.email` — all waitlist signups

The anon key is designed to be public in Supabase's security model. RLS is the
only access control layer, and it is currently set to allow all reads.

**Impact:** Full dump of contractor contact information and waitlist emails.
Violates user privacy expectations and potentially GDPR/CCPA obligations.

**Remediation:**
- Drop `anon_read_notification_channels` policy on `notification_channels`.
- Drop `waitlist_read_anon` policy on `waitlist`.
- Switch `getChannelsForContractor()` in `src/lib/db/notifications.ts:62` from
  `getSupabase()` (anon) to `getSupabaseAdmin()` (service_role).
- Audit any client-side code that reads these tables.

#### AUD-002: No database-level immutability on funded task records

**Severity:** High
**Category:** Data Integrity
**Location:** `supabase/migrations/` (missing trigger), `src/lib/db/tasks.ts`

**Description:**
Once a task is funded on-chain, its `to_human_wallet`, `from_agent_wallet`,
`amount_usdc`, `deadline_unix`, and `payment_request_id` should be immutable.
Today this is enforced only by convention — `updateTaskStatus()` happens to
update only `status` and `updated_at`. However:

- The `service_role` key bypasses RLS entirely.
- No Postgres trigger prevents column mutation on funded tasks.
- No column-level constraints exist.

A future code path, admin script, or Supabase dashboard edit could silently
change the contractor wallet on an active task, redirecting funds upon
completion.

**Note:** The on-chain contract stores `task.worker` immutably, so the actual
USDC transfer destination cannot be altered via Supabase. However, a mismatch
between DB and chain state would cause operational confusion and could mask
an attack.

**Impact:** Medium-term code safety risk. A single misplaced `.update()` call
could corrupt task records.

**Remediation:**
Add a `BEFORE UPDATE` trigger:

```sql
CREATE FUNCTION prevent_task_mutation() RETURNS trigger AS $$
BEGIN
  IF OLD.status NOT IN ('pending') THEN
    IF NEW.to_human_wallet   != OLD.to_human_wallet
    OR NEW.from_agent_wallet != OLD.from_agent_wallet
    OR NEW.amount_usdc       != OLD.amount_usdc
    OR NEW.deadline_unix     != OLD.deadline_unix
    OR NEW.payment_request_id != OLD.payment_request_id
    THEN
      RAISE EXCEPTION 'Cannot modify immutable fields on a funded task';
    END IF;
  END IF;
  RETURN NEW;
END;
$$ LANGUAGE plpgsql;

CREATE TRIGGER trg_prevent_task_mutation
  BEFORE UPDATE ON tasks
  FOR EACH ROW EXECUTE FUNCTION prevent_task_mutation();
```

#### AUD-003: No transaction nonce management for concurrent signer operations

**Severity:** High (mainnet), Low (testnet)
**Category:** Concurrency / Reliability
**Location:** `src/lib/contracts/signer.ts`

**Description:**
The platform signer (`DEPLOYER_PRIVATE_KEY`) is a shared singleton with no
nonce manager, transaction queue, or mutex. When two MCP requests trigger
`completeTaskOnChain()` (or any combination of write operations) concurrently:

1. Both calls invoke `eth_getTransactionCount("pending")`.
2. Both receive the same nonce.
3. One transaction succeeds; the other fails with `nonce too low` or
   `replacement transaction underpriced`.
4. There is no retry logic — the failure propagates as a generic error.

**Impact:** Under concurrent load, legitimate task completions will
intermittently fail. Agents receive opaque on-chain errors with no automatic
recovery.

**Remediation (choose one):**

- **Option A (recommended):** Use viem's built-in `nonceManager` on the account:
  ```typescript
  import { nonceManager } from "viem/accounts";
  return privateKeyToAccount(key as `0x${string}`, { nonceManager });
  ```
- **Option B:** Wrap all signer calls in a `p-queue` with `concurrency: 1` to
  serialize transaction submission.
- **Option C:** Implement retry-with-nonce-bump logic in the catch block.

### Medium

#### AUD-004: `register_notification_channel` echoes PII back to calling agent

**Severity:** Medium
**Category:** Data Privacy
**Location:** `src/lib/mcp/server.ts:447-454`

**Description:**
The MCP tool response includes the full `address` field (email, Telegram ID,
Discord ID, or webhook URL) in the JSON returned to the calling agent:

```typescript
channel: {
  id: channel.id,
  contractor_id: channel.contractor_id,
  type: channel.type,
  address: channel.address,  // PII leaked to agent context
  accepts_auto_booking: channel.accepts_auto_booking,
}
```

While the caller submitted this data, the response goes into the agent's context
window — an LLM that may log, summarize, or relay this information.

**Remediation:** Strip or mask the `address` field in the response. Return
only `id`, `type`, and `accepts_auto_booking`.

#### AUD-005: Task state machine enforced only in application code

**Severity:** Medium
**Category:** Data Integrity
**Location:** `src/lib/db/tasks.ts:78-84`

**Description:**
The `VALID_TRANSITIONS` map prevents illegal status jumps (e.g.,
`completed` -> `active`), but it exists only in TypeScript. A direct
`service_role` query, Supabase dashboard edit, or migration script can set any
status on any task without constraint.

Additionally, `updateTaskStatus()` has a TOCTOU (time-of-check-time-of-use)
race: it reads the current status in one query (line 93), validates the
transition in JS, then issues the update in a separate query (line 109). Two
concurrent calls could both read `active` and both write `completed`.

**Note:** The on-chain contract independently enforces its own state machine
(`TaskState.Funded` -> `TaskState.Completed`), so double-payment is impossible
regardless. This is an operational correctness issue, not a funds-safety issue.

**Remediation:**
- Move the state machine to a Postgres trigger or use an atomic
  `UPDATE ... WHERE status = $current_status` pattern.
- For the TOCTOU race, change the update query to:
  ```sql
  UPDATE tasks SET status = $new, updated_at = now()
  WHERE payment_request_id = $id AND status = ANY($allowed_sources)
  ```
  Check `rowCount === 1` to confirm the transition succeeded.

#### AUD-006: DB/chain state inconsistency on partial failure

**Severity:** Medium
**Category:** Reliability / Data Integrity
**Location:** `src/lib/mcp/server.ts:342-367`

**Description:**
The `confirm_task_completion` flow correctly orders operations: on-chain first,
then DB update. If the on-chain call fails, the DB is not updated (correct).

However, if the on-chain call **succeeds** but `updateTaskStatus()` fails
(Supabase outage, network error), the contract shows `Completed` while the
database still shows `active`. The task appears stuck. A retry attempt would
fail on-chain (`InvalidState(Completed, Funded)`) with a confusing error.

**Impact:** Operational — no fund loss (contract is authoritative), but requires
manual intervention to reconcile.

**Remediation:**
- Add a `completing` intermediate status set before the on-chain call.
- Or implement a periodic reconciliation job that compares on-chain state with
  DB state and fixes mismatches.
- Or wrap the on-chain call + DB update in an idempotent pattern that checks
  on-chain state before deciding whether to update DB.

#### AUD-007: `get_contractor` tool exposes notification channel types

**Severity:** Medium-Low
**Category:** Information Disclosure
**Location:** `src/lib/mcp/server.ts:543-546`

**Description:**
The `get_contractor` MCP tool response includes `notification_channels[].type`,
revealing which communication platforms a contractor uses (email, Telegram,
Discord, webhook). It also exposes the internal database UUID (`id`). The actual
channel addresses are correctly excluded.

**Remediation:** Remove `notification_channels` array from the response.
Return only `accepts_auto_booking` (the boolean already present).

### Low / Informational

#### AUD-008: `human_whitepages` resource passes raw DB result without allowlist

**Severity:** Low
**Category:** Defense in Depth
**Location:** `src/lib/mcp/server.ts:917`

**Description:**
The `human_whitepages` MCP resource returns `humans: all` directly from
`getAllHumans()`. Today `getAllHumans()` selects only safe columns (`wallet`,
`categories`, `rate_usdc`, `availability`, `reputation_score`). But if a column
is added to the query in the future, it would leak through with no field mapping
to catch it.

**Remediation:** Add an explicit `.map()` with an allowlist, matching the
pattern used in `search_whitepages`.

#### AUD-009: `tasks` table fully readable by anon key

**Severity:** Low (if intentional)
**Category:** Information Disclosure
**Location:** `supabase/migrations/003_tighten_rls.sql:15-16`

**Description:**
All task records (descriptions, amounts, wallet pairs) are readable by anyone
with the anon key. This may be intentional for a transparent marketplace, but
should be a conscious decision.

**Remediation:** Confirm intent. If tasks should be private, scope the SELECT
policy or remove it and read via `service_role` only.

#### AUD-010: Owner rotation does not cover in-flight `completeTask` calls

**Severity:** Low (informational)
**Category:** Key Management / Incident Response
**Location:** `contracts/CarbonEscrow.sol:123-136`

**Description:**
The contract inherits OpenZeppelin `Ownable`, providing `transferOwnership()`.
This rotates authority for `resolveDispute()` (which uses `onlyOwner`).
However, `completeTask()` checks `msg.sender == task.agent` — the address that
called `createTask()`. For tasks funded before a key rotation, only the old key
can call `completeTask()`.

In a key compromise scenario, in-flight tasks cannot be completed by the new key.
Recovery path: dispute the task, then `resolveDispute(taskId, true)` with the
new owner to release funds to the worker.

**Remediation:** Document the recovery procedure. Optionally, add an
owner-callable `completeTaskByOwner(taskId)` emergency function to the next
contract version.

## Architecture Review

### What's working well

| Area | Assessment |
|------|-----------|
| **Fund flow integrity** | USDC destination is set on-chain at `createTask()` time and read from contract storage at `completeTask()` time. No Supabase data can redirect funds. |
| **Key management** | `DEPLOYER_PRIVATE_KEY` never committed. Only Hardhat's well-known test key appears in git history. `.env.local` is gitignored. |
| **Contract design** | Non-upgradeable, minimal attack surface. Checks-effects-interactions pattern. `nonReentrant` on all state-changing functions. |
| **MCP authorization** | `confirm_task_completion`, `dispute_task`, and `resolve_dispute` all verify `callerWallet == task.from_agent_wallet`. Dual enforcement: app-layer + on-chain. |
| **Double-completion prevention** | Solidity state machine reverts on second call. Contract is authoritative regardless of app-layer races. |
| **Search results privacy** | `search_whitepages` returns only the 5 approved fields via explicit `.map()`. DB query also selects only those columns. |

### Wallet address flow (critical path)

```
createTask(taskId, worker, amount, deadline)
  └─ task.worker = worker           [set once, immutable on-chain]

completeTask(taskId)
  └─ usdc.safeTransfer(task.worker, task.amount)
                        ^^^^^^^^^^^
                        read from contract storage
                        NOT from Supabase
                        NOT from caller input
```

This is the correct architecture. The on-chain contract is the single source
of truth for fund routing.

### Contract inheritance

```
CarbonEscrow
├── Ownable            (non-upgradeable, @openzeppelin/contracts/)
│   ├── owner()
│   ├── transferOwnership(newOwner)  ← signer rotation
│   └── renounceOwnership()
└── ReentrancyGuard    (non-upgradeable, @openzeppelin/contracts/)
    └── nonReentrant modifier
```

Not upgradeable. No proxy. No `Initializable`. Uses a standard `constructor`.
Once deployed, bytecode is permanent. Mainnet deployment requires a fresh
contract.

|----------|---------|---------|

| `NEXT_PUBLIC_REPUTATION_STAKE_CONTRACT` | *(testnet address)* | *(fresh mainnet deploy)* |

### Code changes required before mainnet

The chain selection logic is already wired — `"mainnet"` branches exist in
`escrow.ts`, `signer.ts`, and `hardhat.config.ts`. No application code changes
are needed for the network switch itself.

However, three code/migration changes are **required before mainnet is safe**:

1. **New Supabase migration:** Drop PII-leaking RLS read policies (AUD-001)
2. **New Supabase migration:** Add task immutability trigger (AUD-002)
3. **Signer change:** Add nonce manager or transaction queue (AUD-003)

### Deployment steps

1. Apply new Supabase migrations to production database.
2. Deploy `CarbonEscrow` to Base mainnet: `npx hardhat run scripts/deploy/escrow.ts --network base`
3. Deploy `ReputationStake` to Base mainnet (if applicable).
4. Set all mainnet env vars.
5. Verify contract on Basescan.
6. Test a full task lifecycle on mainnet with a small amount.
